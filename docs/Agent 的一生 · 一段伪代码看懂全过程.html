The Core Loop

主循环 · 所有概念各就各位

这段主循环可以逐层展开:用顶部 Step 1–6 从最简的「裸 LLM 循环」一步步叠加 harness,绿色高亮本步新增的代码行,配合下方讲解看清每一步多了什么能力(点胶囊或按 ← → 切换)。第 6 步是完整版,仍分三段:① 启动装配(一次性)→ ② while 主循环(每轮一次 LLM 决策)→ ③ 收尾(退出后跑一次)。带颜色的 【概念】 标注可点。

按层高亮
agent_loop.pseudo 
1# 最简循环:此刻只有大脑在转,还没有任何 harness
2function agent_loop(goal):
3    context = [goal]                      # 提供给大脑的全部信息,目前只有任务本身
4    while True:
5        text = LLM(context)               # 大脑唯一的工作:看上下文,决定这一步说什么★ 唯一的「任务决策」LLM 调用
6        context += message("assistant", text)   # 把回复追加进历史,下一轮它才记得自己说过什么
7        if looks_done(text):              # 粗糙的判停:文本里像是说完了,就停下
8            return text

Step 1 / 6裸循环

  • agent 的心脏就是一个 while 循环:把对话历史 context 提供给 LLM,它产出下一句,追加回历史,再进入下一轮。
  • 此刻大脑只会“说”(text),还不能“做”:没有工具、没有记忆、没有治理,纯粹是个会应答的循环。
  • context 是它唯一的记忆:不把回复写回历史,下一轮它就忘了自己说过什么。

智 能(CPU) 扩展 与动作 信息 (CE) 治理 配置

智能 · CPU 1 concepts

LLM智能

位置:text, action = LLM(context, menu)

整段代码里 LLM 只占一行,且只做一件事,即任务决策:看着当前上下文和「可用功能菜单」,决定下一步做什么。它不管记忆、不调工具、不做权限,这些都是 harness 的事。要点:「只占一行」说的是任务决策这一行;压缩(compact)、记忆固化、子 agent 内部其实还会各自调用 LLM。案例:产品级 harness 里这些调用还常分给不同模型——主决策用强模型(如 Claude Code 用 Opus),而压缩摘要、危险动作的自动批准分类、子 agent 等辅助任务交给更小更快的模型(如 Haiku);可见「智能」这一层往往是一组分工的 LLM,而非单一模型。把一颗会胡思乱想的 CPU 包成一台能工作的计算机,正是 harness 的工作。

扩展与动作 4 concepts

Skill扩展与动作

位置:register_skills() · H.skills.load()

一个 SKILL.md(描述 + 指令)。大脑根据描述自己判断该不该用(model-invoked),你不用写 if。妙在渐进式披露:平时只占一张「名片」(description)的上下文,命中了才把 SKILL.md 全文读进上下文,载入的是「指令文本」,不是去运行一个程序;它捆绑的脚本是随后由模型当作普通工具(call_tool)去执行的。这正是 Skill 与 Tool 的分界:Skill 是给大脑读的(提示空间),Tool 是给 harness 执行的(动作空间)。

Tool 原语扩展与动作

位置:register_builtin_tools() · H.sandbox.run()

工具调用是 agent 落地的最底层原语:读写文件、跑 bash、执行脚本。agent 的一切动作最终都 ground 在一个 tool call 上。MCP 只是「暴露工具的协议」,而 Tool 原语是协议运的货本身。顺带一提:代码里把 use_skill / spawn_subagent / take_note 拆成并列分支只为讲清楚;真实实现里它们也都是菜单里的工具:载入 Skill、派子 agent(Task 工具)、记笔记(memory 工具),最终都收敛成一次 tool call。

MCP扩展与动作

位置:connect_mcp_servers() · H.mcp.call()

Model Context Protocol,接外部工具的统一插口。工具方实现一次 MCP server,所有 agent 立刻都能调,接工具从 N×M 的私货变成 N+M 的公共品。它是 agent 世界的 USB / 驱动 ABI。一点边界:MCP 工具在外部执行,本地沙箱覆盖不到它的副作用,治理只能靠事前的 guardrail/审批与协议层信任。(MCP 已于 2025 年捐给 Linux 基金会旗下的 Agentic AI Foundation,OpenAI、Google、微软等均已加入;N+M 的论点不受影响、反而更强。)

子 Agent / Isolate扩展与动作

位置:spawn_subagent → agent_loop(subtask, child_H)

派发一个独立上下文的子 agent,本质就是递归调用 agent_loop 自己。子任务在隔离的上下文里跑完,只把「摘要」带回主流程,不污染主上下文。这是 CE 四件事里的 Isolate,也是多 agent 协作的基本形态。注意:递归写法只体现了「隔离」;子 agent 的另一半价值是「并行」,可同时 fan-out 多个。实践中子 agent 通常拿一个受限的 harness(不同提示词、工具子集、甚至更小的模型),并需设递归深度 / 总预算上限以防 fork 炸弹。

信息 · CE 4 concepts

CE 上下文工程信息

位置:context = H.system_prompt + H.rules + …(开局装配那几行)

Context Engineering,组织「给大脑什么信息」,是 harness 与执行/扩展并列的另一半工作。四件核心操作:Write(写出去/读回来)、Select(检索)、Compress(压缩)、Isolate(拆分)。其中 Write 不止即时记笔记,也包括把结构化任务清单(如 Claude Code 的 TodoWrite)持续回注上下文,在长任务里维持目标方向、对抗偏题。若说 LLM 是 CPU,上下文窗口就是它的 RAM,而 CE 就是这台机器的内存管理。决定任务质量的,始终是放入上下文窗口的信息质量。

记忆 (Write)信息

位置:H.memory.read / write(循环内) / consolidate(会话末)

LLM 每次对话从零开始,记忆是 harness 替它补的「跨会话长期记忆」。它有两个发生时机:循环内,大脑主动决定「这条值得记」时写 scratchpad;会话末,consolidate 把零散笔记固化成长期记忆。现代实现里,读和写常常都是模型自己调用的工具(read_memory / write_memory),而非全由 harness 代劳。对应 scratchpad(会话内) vs 长期记忆(跨会话)的区分。

RAG / Select信息

位置:H.retriever.retrieve(goal)

Retrieval-Augmented Generation,把与当前任务相关的外部知识检索出来、纳入上下文,给模型事实锚点。它是 CE 四件事里的 Select:不是把所有信息放进去,而是只检索「这一步需要的恰到好处的信息」。检索有两个时机:开局一次性预检索(代码里这行),以及更现代的 just-in-time,模型在循环内用搜索/读文件工具按需取数(Claude Code 即用 glob/grep 即时检索),二者常混合使用。

熵管理 / Compress信息

位置:compact(context)

上下文窗口会随多轮累积而膨胀、被无关内容稀释(上下文腐化)。熵管理在窗口将满时压缩 / 裁剪,保住高信号 token。它是 CE 四件事里的 Compress,主动的工程工作,而非被动等窗口被塞满。

治理 5 concepts

Hooks治理

位置:register_hooks() · run_hooks(Pre/Post/SessionEnd)

绑定到生命周期事件(执行前 PreToolUse、执行后 PostToolUse、会话开始/结束)的脚本。它让某些检查「自动发生」而无需提示,就像 Git hooks,只不过是给 Agent 用的。它不止能旁观:PreToolUse 钩子可以返回 deny 直接拦下这次工具调用,所以它和 Guardrail 常是同一套闸的两种用法(一个「拦」,一个「触发」)。权限拦截、格式检查、密钥扫描常由 Hook 实现。

架构约束 / Sandbox治理

位置:create_sandbox() · H.sandbox.run()

给 Agent 的每一次执行加上隔离边界与资源上限(CPU、内存、超时、文件系统范围)。Sandbox 是「架构约束」最硬的一种:类型系统、目录边界、容器隔离,本质都是用结构限制 Agent 的行动空间,把风险关进笼子。

可观测性 / Tracing治理

位置:H.tracer.record(每轮) · H.tracer.flush(会话末)

记录 Agent 的每一步决策与动作,事后可回放、可审计、可调试。注意它的双节奏:record 在循环内每轮都跑,而且记的是「决策 + 观察结果」成对的轨迹,否则只剩决策、回放不出究竟发生了什么;flush 在会话末最终落盘。真实系统还会循环内增量落盘,否则中途崩溃就丢了整条决策链。此外,产品级 harness 常在每步对文件状态另做快照(如 git write-tree),工具失败时回滚到上一快照、防止仓库被改坏——这是与决策轨迹互补的「状态快照 / checkpoint」,OpenCode、Claude Code 都有。

权限门控 / Guardrail治理

位置:H.guardrail.allow(action)

在动作执行「之前」做策略检查,把危险动作先拦下来。高风险场景(删库、转账、对外发布)在这里停下,转人工审批(Human-in-the-Loop)。它和 Hooks 常是同一套机制的不同用法:Guardrail 是「拦」,Hooks 是「触发」。

反馈循环治理

位置:run_sensors(tests, lint, types, CI)

这是 Harness Engineering 的灵魂,它有两半。最基础的一半:工具调用本身失败时,把错误信息回灌进上下文,模型据此改换路径重试,失败结果同样是环境给的客观事实。更主动的一半:Agent 改完代码后自动跑测试 / Lint / 类型检查 / CI,把失败回灌、据此自我修正。传统软件工程二十年的质量门禁,在 Agent 时代获得第二个用途:不再只是给人的门禁,而是给机器的护栏(这一「机器护栏」的提法见 Kent Beck 关于 augmented coding 的论述)。案例:OpenCode 把 LSP 实时诊断作为最即时的反馈源——edit 一完成就立刻拿到类型 / 语法 / 未定义变量的报错并回灌,比跑完整测试更快闭合循环。

配置 1 concepts

规则文件配置

位置:load_rule_files(CLAUDE.md, AGENTS.md)

CLAUDE.md / AGENTS.md,启动时自动加载进上下文的声明式约束。它把「每次都要叮嘱的话」(技术栈、代码风格、禁改目录)固化成文件、纳入版本控制。这是最轻的一层 harness,只能做声明式约束,做不了流程编排(那要上 Skill 或代码框架)。

Execution Timeline

执行流程 · 谁跑一次,谁每轮都跑

同一段代码,有的部分一次性执行,有的每轮循环都执行,有的等循环退出才执行一次。这张时间线把三种节 奏分清。

① 一次性

启动装配  build_agent()

跑在 while 之前,只跑一次。把 harness 的零件登记好:规则文件、Skill 名片、内建工具、MCP 连接、Hooks、Sandbox、Tracer。其中 Skill、MCP、Hooks、命令、子 agent 这类可分发的扩展,可整包来自一个 Plugin。

↓
② 每轮重复

while 主循环  while not done and steps < MAX_STEPS

每一轮 = 一次 LLM 决策 + 一次动作执行。下面这些步骤每轮都重新走一遍:

熵管理·压缩列出功能菜单LLM 任务决策(唯一一行)决策入上下文权 限守卫分发:Skill / 工具 / 子AgentHooks 前后触发记轨迹反馈循环回灌
↓退出条件 = done==True(任务完成) 或 steps==MAX_STEPS(预算耗尽)
③ 退出后一次

收尾  memory.consolidate() · tracer.flush()

循环退出后,跑且只跑一次:把循环内零散记下的笔记固化成长期记忆,把整条决策链最终落盘,返回答案。注意:这里的「记忆/落盘」各自还有一半发生在循环内 (即时记笔记、每轮记轨迹),收尾只是「会话末的固化」。

Coda · Loop Engineering

尾声 · 循环之上还有循环

本页的 while 循环叫 agent loop,但 2026 年中的热词 Loop Engineering(Addy Osmani 命名)说的不是它,而是循环之上的那一层:到目前为止,goal 仍是人给的,人守在屏幕前等它返回。Loop Engineering 把这最后一环也工程化——谁启动循环、谁验收结果、谁决定下一轮做什么。Claude Code 负责人 Boris Cherny 的表述是:「I don't prompt Claude anymore. I have loops running that prompt Claude and figuring out what to do. My job is to write loops.」一点提醒:这个词 2026 年 6 月才诞生,内涵与边界仍在快速演变,Osmani 本人也明说仍持保留态度——把它当作「正在成形的概念」,而非定论。

outer_loop.pseudo 
1# ---------- Loop Engineering:人退到循环之上 ----------
2while True:
3    goal   = find_work(backlog, issues, schedule)   # 系统自己找活:issue 队列 / 定时任务 / CI 事件,不再等人给 goal
4    H      = build_agent(project)                   # 每个任务装配一次 harness:即本页 ① 启动装配
5    answer = agent_loop(goal, H)                    # 本页讲的全部内容,在这里只占一行★ 整页 = 这一行
6    verify_and_record(answer, external_state)       # 客观验收(测试 / CI),写回外部状态,决定下一轮
7 
8# ⚠ 注意:以上是教学用的极端简化版。真实的外层循环至少还需要:
9#   ① 预算与停机守卫(内循环有 MAX_STEPS,外循环同样不能裸奔)   ② verify 连续失败 N 次后 escalate_to_human,升级给人
10#   ③ 多 agent 并行时用 worktree 隔离工作区,防文件冲突   ④ goal 与进度写回外部状态(issue / STATE 文件):agent 会忘,repo 不会

Harness × Loop两层分工,一个公式

  • Harness Engineering(本页全部内容)回答空间问题:任意一个瞬间,模型周围有什么——工具、沙箱、权限门控、反馈传感器、可观测性。它决定循环体内每一行「能做什么、被什么拦着」。
  • Loop Engineering(上面六行)回答时间问题:圈与圈之间如何演进——谁触发循环、用什么客观信号验收、何时停机、失败后是重试还是升级给人。它决定这些行「以什么控制流被反复执行」。注意别与本页的 agent loop 混为一谈:agent loop 是循环体本身,loop engineering 是循环之上的编排。
  • 谱系:Prompt Engineering (2023) → Context Engineering (2025) → Harness Engineering (2026.2) → Loop Engineering (2026.6)。公式分两层写更稳:单次 Agent = Model × Harness;长驻 Agentic System = Agent + Outer Loop。本页开头那句「LLM 只占一行」在外层同构地再现:整个 agent_loop 也只占一行——你以为它是主程序,其实它是别人循环体里的一行。

大脑只做一行任务决策,其余全是工程。裸 LLM 是一块发烫的硅;harness 把它变成一台计算机;MCP 与 Plugin 再把这台计算机变成一个生态;而 loop engineering 让这台计算机自己开机干活。
